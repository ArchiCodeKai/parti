## 一、要解決的問題

中文世界缺少一個**有策展觀點的現代建築知識入口**：

- Wikipedia 資訊平鋪，沒有觀點、沒有關係結構、視覺與建築無關。
- ArchDaily 是新聞流，適合追新案，不適合建立知識譜系。
- 教科書有譜系但是死的 — 不可探索、不可比較、不會回應你。
- ChatGPT 能回答單點問題，但給不出「柯比意、密斯、萊特三人構成什麼樣的三角關係」
  這種**結構性**的知識呈現。

PARTI 的答案：**用幾何本身呈現建築知識的結構。**
parti 是建築學術語 — 設計的初始概念、基礎構思。這個網站的 parti 就是：
知識條目是點，關係是線，流派是面；人數（N）決定幾何形式
（1 同心圓 / 2 中垂線 / 3 三點定圓 / 4–6 Delaunay / 7–12 Voronoi / 13+ 力導向圖）。

## 二、目標使用者

| 優先級 | 族群 | 需求 | 對應功能 |
|---|---|---|---|
| 主要 | 建築系學生、研究生、備課教師 | 快速建立譜系感、查人物/作品/流派關係 | 詳情頁、Compare、Cmd+K |
| 次要 | 對建築有興趣的設計圈與一般讀者 | 低門檻探索、被視覺吸引而留下 | Landing、四大入口、地圖 |
| 曝光 | Awwwards / 設計社群評審 | 看到工藝與原創互動 | Landing 材質卷軸、幾何語言、Hidden-until-Touched |

**維護者現實約束（一人 side project）**：所有範圍決策都以「一個人能維護」為前提。
這是下方大量「不做」的根本原因。

## 三、產品原則

1. **策展，不聚合。** 每個字都是編輯過的觀點，不是爬來的資料。
2. **幾何即論述。** 視覺不是裝飾，是知識結構的直接呈現。禁止裝飾性插畫。
3. **預設冷靜，互動才爆色。** Glass Editorial + Hidden-until-Touched，朱紅 ≤ 3%。
4. **內容深度優先於數量。** 29 條寫滿規格字數，勝過 200 條 stub。
5. **對外數字必須誠實。** 條目數一律由資料長度導出，禁止寫死宣傳數字。

### 4.3 v1 成功標準

- 29 + 68 + 27 條目全部達到規格字數，可瀏覽、可搜尋。

- 桌機 Lighthouse Performance > 90。
- 375px 寬所有頁面不破版。
- 投稿 Awwwards。

## 五、明確不做什麼

### 永久不做（與定位衝突）

- **使用者編輯條目** — 策展權不外放。
- **AI 聊天機器人** — 會稀釋觀點；PARTI 的價值恰是「非生成式」的策展。
- **後端資料庫 / 帳號 / 登入** — 純前端靜態資料，內容進版本控制。
- **CAD 圖重繪、建築構造教學** — 版權與工作量爆炸，且與策展氣質不合。
- **寫實地圖圖磚（Mapbox）** — 已定案純 SVG 幾何座標場，與設計哲學一致。

### v1 不做（延至 v2+，有意識地推遲）

- 條目擴量至 200+ / 600+（v1 鎖 29/68/27，見產品原則 4）
- school / firm / book / exhibition / award / theorist 六種 Entity
- 條目級 OG image、書籤收藏、暗色模式、閱讀進度、Surprise me
- 中英雙語切換（介面已中英混排，真正的 i18n 留 v3）
- 使用者留言（v3 再議，且必須人工審核）
- N ≥ 13 的力導向圖（v1 資料規模到不了 13 人同流派的場景）

## 六、v2 方向（僅列航向，不展開）

v1 ship + 投獎後才啟動：內容擴量（優先 Pritzker 補齊至歷屆全員）、
Compare 分享 URL（?ids=）、條目級 OG、六種新 Entity、暗色模式。
規格屆時另立文件，不預先設計。
